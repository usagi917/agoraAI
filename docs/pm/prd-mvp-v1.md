## 1. 概要

### プロダクト名と一行説明

**Agent AI v1.0** — 業界レポートやメモを入力するだけで、多視点の構造化シナリオ分析レポートを数分で自動生成するAIツール。

### ビジョンステートメント

経営コンサルタントがシナリオ分析の初稿作成に費やす時間を「数週間」から「数時間」に短縮し、より高付加価値な戦略提言に集中できる環境を実現する。マルチエージェントシミュレーションにより、単一視点では見落とされがちなリスク・機会を構造的に可視化する。

### 成功の定義

MVP リリースから3ヶ月以内に以下を達成する:

1. **パイロット顧客3社**が本番業務でAgent AIを使用している
2. **レポート生成時間**: 入力から最終レポート出力まで平均10分以内
3. **LLM APIコスト**: 1実行あたり平均$3以下（上限$5）
4. **ユーザーリテンション**: パイロット顧客のマネージャー層が月2回以上利用

## 2. 問題定義

### ターゲットユーザーの現在の課題

#### シナリオ1: 業界分析レポートの初稿作成

田中マネージャー（コンサルファーム、在籍8年）は、クライアント向けの自動車業界シナリオ分析レポートを作成している。業界レポート5本、社内メモ3本、ニュース記事10本を読み込み、PESTLE分析フレームワークに沿って3つのシナリオ（楽観・中立・悲観）を構築する必要がある。

**現状**: テキストを手作業で読み込み、Excel/PowerPointに転記しながら構造化するのに**2〜3週間**かかる。分析視点はアナリストの経験に依存し、見落としリスクが高い。

#### シナリオ2: クライアントレビュー前の追加分析

パートナーの鈴木氏がクライアントへのプレゼン前日に「規制強化シナリオも追加して」と依頼。田中マネージャーは徹夜でシナリオを追加するが、既存シナリオとの整合性チェックが不十分なまま提出することになる。

**現状**: 追加シナリオの作成に**2〜3日**。既存分析との整合性は人力で確認するため、矛盾が残るリスクがある。

#### シナリオ3: 複数プロジェクトの同時進行

田中マネージャーは同時に3件のプロジェクトでシナリオ分析を進行中。各プロジェクトで類似した「市場参入分析」を行っているが、業界や地域が異なるため、テンプレートの再利用が困難。

**現状**: 類似分析を毎回ゼロから作成。ナレッジの再利用率が低い。

### 課題の深刻度と頻度

| 課題 | 深刻度 | 頻度 |
|------|--------|------|
| シナリオ分析の初稿作成に時間がかかりすぎる | 高 | 月2-3回/マネージャー |
| 分析視点の偏り・見落とし | 高 | ほぼ毎回 |
| 追加シナリオ要求への対応遅延 | 中 | 月1-2回 |
| ナレッジの再利用が困難 | 中 | 常時 |
| レポートのフォーマット統一が難しい | 低 | 常時 |

### 現在の代替手段とその限界

| 代替手段 | 利点 | 限界 |
|----------|------|------|
| **手作業（Excel/PPT）** | 柔軟性が高い | 2-3週間かかる、視点の偏り |
| **ChatGPT/Claude直接利用** | 速い | 構造化されない、再現性がない、文脈が1セッションで消える |
| **McKinseyなどの既存フレームワークツール** | 方法論が確立 | 自動化されていない、カスタマイズ困難 |
| **BIツール（Tableau等）** | データ可視化に強い | シナリオ「生成」はできない、定量データ前提 |

## 4. ユーザーストーリー & ジョブストーリー

### Must-have（P0）

#### US-01: テキスト入力によるシミュレーション開始

> **ユーザーとして**、テキストまたはMarkdown形式の業界レポート・メモを入力して、シミュレーションを開始したい。
> **それにより**、構造化された分析レポートの初稿を自動で得られる。

**受け入れ基準:**
- テキストエリアに最大50,000文字のテキストを入力できる
- 「分析開始」ボタンを押すとシミュレーションが開始される
- 入力が空の場合、エラーメッセージが表示される
- `business_analysis` テンプレートがデフォルトで選択されている

#### US-02: リアルタイム進捗表示

> **ユーザーとして**、シミュレーションの進捗をリアルタイムで確認したい。
> **それにより**、処理の透明性を感じ、完了までの待ち時間を予測できる。

**受け入れ基準:**
- SSEにより以下のイベントがリアルタイムに表示される: `world_initialized`, `agents_built`, `round_completed`(各ラウンド), `report_section_done`(各セクション), `run_completed`
- 進捗バーが全体の何%完了かを表示する
- 各ラウンドの完了時にイベントサマリーが表示される

#### US-03: 構造化レポートの閲覧

> **ユーザーとして**、生成されたレポートをセクション単位で閲覧したい。
> **それにより**、必要な箇所だけを素早く確認・抽出できる。

**受け入れ基準:**
- レポートは以下のセクション構成で表示される: エグゼクティブサマリー、前提条件、エンティティ概要、シミュレーション要約、シナリオ比較、イベントタイムライン、リスク、機会、推奨アクション、不確実性、エビデンス参照
- 各セクションは折りたたみ/展開が可能
- レポート全文をMarkdown形式でエクスポートできる

#### US-04: 2Dグラフ可視化

> **ユーザーとして**、エンティティ間の関係性を2Dグラフで視覚的に確認したい。
> **それにより**、ステークホルダー関係を直感的に把握できる。

**受け入れ基準:**
- ノード（エンティティ）とエッジ（関係）が2Dネットワークグラフとして表示される
- ノードはエンティティタイプ別に色分けされる
- エッジの太さは関係の重みを反映する
- ラウンド別にグラフの変化を確認できる（テンポラルリプレイ）

#### US-05: Follow-up質問

> **ユーザーとして**、生成されたレポートに対してFollow-up質問を投げたい。
> **それにより**、特定のシナリオやリスクについて深堀り分析を得られる。

**受け入れ基準:**
- レポート閲覧画面にFollow-up質問の入力欄がある
- 質問を送信すると、既存のレポートとworld_stateをコンテキストとして回答が生成される
- 過去のFollow-up質問と回答の履歴が閲覧できる

#### US-06: ユーザー認証

> **ユーザーとして**、メールアドレスとパスワードでログインしたい。
> **それにより**、自分のシミュレーション履歴とレポートに安全にアクセスできる。

**受け入れ基準:**
- メールアドレス + パスワードによるサインアップ/ログインが可能
- 未認証ユーザーはAPI/UIにアクセスできない
- JWTトークンベースのセッション管理
- パスワードはbcryptでハッシュ化されて保存される

#### US-07: シミュレーション履歴

> **ユーザーとして**、過去に実行したシミュレーションの一覧を確認し、レポートを再閲覧したい。
> **それにより**、過去の分析結果を参照・比較できる。

**受け入れ基準:**
- シミュレーション一覧が新しい順に表示される
- 各シミュレーションのステータス（完了/実行中/失敗）が表示される
- 完了したシミュレーションのレポートをクリックで再閲覧できる

### Should-have（P1）

#### US-08: PDF入力対応

> **ユーザーとして**、PDFファイルをアップロードして入力としたい。
> **それにより**、手動でのテキストコピーが不要になる。

**受け入れ基準:**
- PDFファイル（最大20MB）をアップロードできる
- PyMuPDFでテキストが自動抽出される
- 抽出されたテキストのプレビューが表示される
- 複数PDFの同時アップロードが可能

#### US-09: GraphRAG Entity/Relation抽出

> **ユーザーとして**、長文ドキュメントでもエンティティと関係を漏れなく抽出してほしい。
> **それにより**、8,000文字制限による情報欠落を回避できる。

**受け入れ基準:**
- GraphRAGパイプライン（Entity抽出 + Relation抽出 + 重複解決）が動作する
- Community検出はMVP後（Entity/Relationのみ）
- 抽出されたエンティティ数と関係数がSSEで通知される

#### US-10: 複数テンプレート対応

> **ユーザーとして**、分析目的に応じてテンプレートを選択したい。
> **それにより**、政策シミュレーションやシナリオ探索など目的に特化した分析ができる。

**受け入れ基準:**
- `business_analysis`, `policy_simulation`, `scenario_exploration` の3テンプレートが選択可能
- テンプレートごとにプロンプトとレポートセクション構成が異なる

#### US-11: コスト追跡ダッシュボード

> **管理者として**、LLM APIの使用コストをシミュレーション単位で確認したい。
> **それにより**、コスト最適化の判断材料を得られる。

**受け入れ基準:**
- シミュレーション別のトークン使用量と推定コストが表示される
- 月次の累計コストグラフが表示される
- 1実行あたりのコストが$5を超えた場合に警告が表示される

#### US-12: シミュレーション再実行

> **ユーザーとして**、同じ入力で再度シミュレーションを実行して結果のばらつきを確認したい。
> **それにより**、分析結果のロバスト性を評価できる。

**受け入れ基準:**
- 完了済みシミュレーションに「再実行」ボタンがある
- 同じ入力・テンプレート・プロファイルで新しいシミュレーションが作成される
- 元のシミュレーションとの関連が記録される

```
┌─────────────────────────────────────────────────┐

│              API Gateway / LB                    │
│         (Cloud Run / ECS / Fly.io)               │
└────────────────────┬────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────┐
│            Backend (FastAPI + Uvicorn)            │
│                                                  │
│  ┌─────────────┐  ┌──────────────┐  ┌────────┐  │
│  │ Simulation  │  │ World Builder│  │  SSE   │  │
│  │ Dispatcher  │  │              │  │Manager │  │
│  │ (single     │  │ Document →   │  │        │  │
│  │  mode only) │  │ WorldState   │  └────────┘  │
│  └──────┬──────┘  └──────────────┘              │
│         │                                        │
│  ┌──────▼──────┐  ┌──────────────┐  ┌────────┐  │
│  │ Simulator   │  │   Report     │  │ Cost   │  │

   ┌──────────▼─────┐  ┌─────▼──────┐
```

**MVPで使わないコンポーネント（コードは存在するが無効化）:**
- `SwarmOrchestrator` / `ColonyFactory`（swarm/hybridモード）
- `CognitiveAgent` / `BDIEngine` / `TheoryOfMind`（advancedモード）
- `DebateProtocol`（構造化議論）
- `CommunityDetector`（GraphRAGコミュニティ検出）
- `GameMaster`（advancedモード用ラウンド制御）

### ユーザーフロー（入力 → 処理 → 出力）

```
1. ログイン
   └→ メール+パスワード認証 → JWTトークン発行

2. 新規分析作成
   └→ テキスト/Markdown入力（50,000文字まで）
   └→ テンプレート: business_analysis（MVP固定）
   └→ プロファイル: standard（4ラウンド）
   └→ モード: single（MVP固定）
   └→ 「分析開始」クリック

3. シミュレーション実行（バックグラウンド、SSEで進捗通知）
   ├→ [Phase] World Building: 入力テキスト → エンティティ + 関係抽出 → WorldState構築
   ├→ [Phase] Agent Generation: WorldStateからエージェント（企業/規制機関等）を自動生成
   ├→ [Phase] Round 1-4: 各ラウンドでエージェントが意思決定、イベント発生、関係変化
   ├→ [Phase] Graph Projection: 各ラウンド後にグラフスナップショット保存
   └→ [Phase] Report Generation: 11セクションの構造化レポート生成

4. 結果閲覧
   ├→ レポート（セクション別閲覧、Markdownエクスポート）
   ├→ 2Dグラフ（ラウンド別テンポラルリプレイ）
   ├→ タイムライン（イベント時系列表示）
   └→ Follow-up質問（追加分析の生成）
```

### 技術的な選択とその理由

| 技術選択 | 理由 |
|----------|------|
| **FastAPI + Uvicorn** | 非同期I/Oに最適、SSEネイティブサポート、型安全なAPI定義 |
| **PostgreSQL (asyncpg)** | 本番環境での信頼性、JSON型カラムでスキーマレスデータも格納可能 |
| **SSE (Server-Sent Events)** | WebSocketより実装がシンプル、ファイアウォール/プロキシ透過性が高い、サーバー→クライアント単方向で十分 |
| **LiteLLM** | OpenAI互換APIの抽象化、将来的なマルチプロバイダ対応が容易 |
| **gpt-4o + gpt-5-nano 2層戦略** | 高品質タスク(world_build, report)にgpt-4o、ラウンド処理等にgpt-5-nanoでコスト最適化 |
| **Docker + uv** | 再現性の高いビルド、uvによる高速な依存解決 |
| **Next.js (フロントエンド)** | SSR/SSG対応、React エコシステム活用、TypeScript |

## 6. 機能要件

### P0: MVP Core

#### F-01: テキスト/Markdown入力

| 項目 | 仕様 |
|------|------|
| 入力方式 | テキストエリア（Markdown対応） |
| 最大文字数 | 50,000文字 |
| バリデーション | 空入力不可、最小100文字 |
| エンコーディング | UTF-8 |
| API | `POST /api/simulations` の `prompt_text` フィールド |

#### F-02: WorldState構築

| 項目 | 仕様 |
|------|------|
| 入力 | ドキュメントテキスト + テンプレートプロンプト |
| 出力 | entities[], relations[], world_summary, timeline |
| LLMタスク | `world_build` (gpt-4o, temperature=0.3) |
| テキスト上限 | 8,000文字（GraphRAG未使用時） |
| バリデーション | `validate_world_build` によるJSON構造検証 |
| リトライ | 最大3回（`call_with_retry`） |
| 実装 | `src/app/services/world_builder.py` の `build_world()` |

#### F-03: エージェント自動生成

| 項目 | 仕様 |
|------|------|
| 入力 | WorldState + テンプレートのagent_generateプロンプト |
| 出力 | agents[]（名前、役割、目標、戦略、意思決定パターン） |
| LLMタスク | `agent_generate` (gpt-5-nano, temperature=0.5) |
| 実装 | `src/app/services/agent_generator.py` |

#### F-04: シミュレーション実行（Legacyモード、4ラウンド）

| 項目 | 仕様 |
|------|------|
| モード | legacy（`cognitive_mode: legacy`に固定） |
| 実行モード | single（`mode: single`に固定） |
| ラウンド数 | standard プロファイル = 4ラウンド |
| 各ラウンドの処理 | エージェント意思決定 → イベント生成 → WorldState更新 → グラフ投影 |
| LLMタスク | `round_process` (gpt-5-nano, temperature=0.5) |
| プロファイル | preview(2), standard(4), quality(6) |
| 実装 | `src/app/services/simulator.py` の `SingleRunSimulator` |

#### F-05: レポート生成

## 10. リスクと緩和策

### 技術的リスク

| リスク | 影響 | 発生可能性 | 緩和策 |
|--------|------|-----------|--------|
| **LLM APIの品質変動** | レポート品質のばらつき | 高 | `call_with_retry` で最大3回リトライ、`validate_world_build` でJSON構造検証、temperatureチューニング |
| **LLM APIのレート制限** | シミュレーション遅延・失敗 | 中 | `rate_limiter.py` で RPM/TPM 制御（現在 500RPM, 200K TPM）、`gpt-5-nano` を多用してTier1呼び出しを最小化 |
| **LLM APIコスト超過** | 予算オーバー | 中 | `cost_tracker.py` でリアルタイム記録、実行前の推定コスト表示、月間予算アラート |
| **SSE接続の不安定性** | 進捗表示の途切れ | 中 | クライアント側自動再接続、最終結果はDB永続化されているためポーリングでフォールバック |
| **大規模テキスト入力時のトークン超過** | WorldState構築の情報欠落 | 高 | Phase 1では8,000文字制限、Phase 2でGraphRAG有効化により全文カバレッジ実現 |

### ビジネスリスク

| リスク | 影響 | 発生可能性 | 緩和策 |
|--------|------|-----------|--------|
| **レポート品質がコンサル基準に達しない** | パイロット顧客の離脱 | 高 | 「初稿ツール」としてのポジショニング（最終レポートではない）、パイロット期間中の密なフィードバックループ |
| **機密データ漏洩への懸念** | 導入拒否 | 高 | OpenAI API のデータ利用ポリシーの明確化、将来的にAzure OpenAI Service（データ処理がAzure内で完結）への移行オプション提示 |
| **競合ツールの出現** | 差別化困難 | 中 | マルチエージェント×グラフという独自性の維持、Phase 3のBDI/SwarmMindが参入障壁 |
| **価格設定の失敗** | 収益化困難 | 中 | パイロット期間で支払い意思額（WTP）を調査、コストベースの下限設定 |

### 運用リスク

| リスク | 影響 | 発生可能性 | 緩和策 |
|--------|------|-----------|--------|
| **OpenAI API障害** | サービス停止 | 低 | LiteLLMによるプロバイダ切替（Anthropic Claude等）への将来対応、障害時はUIにメンテナンスメッセージ |
| **DBデータ肥大化** | ストレージコスト増、パフォーマンス低下 | 中 | 90日データ保持ポリシー、世界状態のJSON圧縮、古いシミュレーションのアーカイブ |
| **少人数チームでの運用負荷** | バグ対応遅延 | 高 | 監視自動化（ヘルスチェック、エラーアラート）、E2Eテスト自動化、ドキュメント整備 |

## 11. 付録

### 用語集

| 用語 | 定義 |
|------|------|
| **WorldState** | シミュレーション内の「世界」を表すデータ構造。エンティティ（組織、人物、市場等）とその関係性を含む |
| **エンティティ (Entity)** | WorldState内の主体。企業、規制機関、消費者グループ等 |
| **リレーション (Relation)** | エンティティ間の関係性。競合、協力、規制、供給等 |
| **エージェント (Agent)** | シミュレーション内で意思決定を行う自律的な主体。各エンティティに対応 |
| **ラウンド (Round)** | シミュレーションの1ステップ。各ラウンドでエージェントが行動し、WorldStateが更新される |
| **Legacyモード** | LLMに直接プロンプトを送信してラウンドを処理するシンプルなモード（MVP対象） |
| **Advancedモード** | BDI認知アーキテクチャ + GameMaster によるラウンド処理（Phase 3対象） |
| **BDI (Belief-Desire-Intention)** | エージェントの認知アーキテクチャ。信念・欲求・意図に基づく意思決定モデル |
| **SwarmMind** | 複数のColony（エージェント群）を並列実行し、多視点の結果を統合する仕組み |
| **Colony** | SwarmMindの1単位。固有の視点（楽観/悲観/規制強化等）を持つエージェント群 |
| **GraphRAG** | 文書からナレッジグラフを構築し、RAG（Retrieval Augmented Generation）に活用する手法 |
| **SSE (Server-Sent Events)** | サーバーからクライアントへのリアルタイムイベント配信プロトコル |
| **テンプレート** | 分析の種類ごとに定義されたプロンプト集。world_build, agent_generate, round_process, report_sectionsを含む |
| **プロファイル** | 実行品質の設定。preview(2ラウンド), standard(4ラウンド), quality(6ラウンド) |
| **Theory of Mind (ToM)** | エージェントが他エージェントの信念・意図を推論する機能（Phase 3対象） |
| **Debate Protocol** | エージェント間の構造化された議論プロトコル（Phase 3対象） |
